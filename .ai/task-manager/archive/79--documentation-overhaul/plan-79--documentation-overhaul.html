Executive Summary

The current documentation site has grown organically and suffers from several problems: inconsistent terminology, scattered focus across too many pages, and a messaging strategy that undersells the core value proposition. The docs explain features but fail to clearly articulate why this tool produces better results than plan mode and how it achieves that through context isolation.

This overhaul will restructure the entire docs/ site around four pillars: (1) zero-tooling simplicity — everything is plain text files and Agent Skills, (2) multi-harness portability, (3) superior results vs plan mode through architectural advantages, and (4) context management via clean-context sub-agents. A formal glossary will be established both in the docs and in AGENTS.md to create a shared language: work order, plan, execution blueprint, and phases.

The restructured site will have fewer, more focused pages. Redundant content currently spread across architecture.md, features.md, and comparison.md will be consolidated. The navigation hierarchy will flatten — every important page should be reachable within one click from the homepage.

Terminology Standardization

Objective: Establish and enforce a canonical vocabulary across all documentation pages and AGENTS.md.

The following terms must be used consistently throughout all docs:

  • Work order — The user's request describing what they want accomplished. Input to the system.
  • Plan — The comprehensive document the LLM produces by refining the work order. Covers requirements, architecture, risks, success criteria. Output of Phase 1.
  • Execution blueprint — The structured output of task generation (Phase 2). Contains all tasks organized into phases with dependency mappings. Replaces the informal "task list" language.
  • Phase — A group of tasks within the execution blueprint. Tasks within a phase execute in parallel. Phases execute in sequence. This is the unit of parallelism.
  • Task — An atomic unit of work within a phase. Has 1-2 skills, acceptance criteria, and dependencies. Executed by a sub-agent with clean context.
  • Sub-agent — A specialized AI agent that executes a single task with focused, clean context. Not the main conversation agent.

Terms to retire from docs: "progressive refinement" (replace with "three-phase workflow"), "validation gates" (replace with "review gates" when referring to human review, keep "quality gates" for automated checks), "cognitive overload" (replace with concrete explanations of context management).

Four Messaging Pillars

Objective: Reframe all documentation around four clear value propositions that the user identified.

Pillar 1: Zero Tooling — AI Task Manager requires no additional tools, API keys, or services. Everything is plain text files (Markdown, YAML frontmatter) and Agent Skills. Plans, tasks, and configurations are human-readable and git-friendly. Works within existing AI subscriptions. This contrasts with API-based tools like Plandex, Conductor Tasks, and Claude Task Master that require separate API keys and pay-per-token pricing.

Pillar 2: Multi-Harness Support — Skills are harness-agnostic. A single SKILL.md works for every harness that supports the Agent Skills format. Team members can use different harnesses (Claude Code, Gemini CLI, GitHub Copilot, Codex, Cursor, OpenCode) while collaborating on the same plans. Initialize once, install skills per developer. The shared workspace (.ai/task-manager/) is the collaboration point.

Pillar 3: Better Results Than Plan Mode — Plan mode in any AI coding assistant creates and executes in a single context, leading to scope creep, diluted focus, and quality degradation as context grows. AI Task Manager separates planning from execution, introduces human review gates between phases, enforces YAGNI through multiple checkpoints, and deploys specialized sub-agents per task. The result is more focused, higher-quality implementations.

Pillar 4: Context Management — This is the architectural core. Each phase operates with clean, focused context. Plan creation gets maximum reasoning power focused solely on requirements and architecture. Task generation gets a clean context loaded only with the approved plan. Execution deploys sub-agents that each receive only their specific task, its dependencies, and acceptance criteria — not the entire conversation history. This prevents the quality degradation that occurs when a single context accumulates all planning, decomposition, and implementation concerns. Sub-agents can execute in parallel (within a phase) or in sequence (across phases), depending on task dependencies.

Risk Considerations and Mitigation Strategies

Technical Risks
  • Broken internal links after page restructuring: Renaming and deleting pages will break cross-references throughout the site.
    • Mitigation: Each task that writes/modifies a page must update all internal links across all docs pages. A final validation task should grep for broken references.
  • Jekyll navigation breaks with flat hierarchy: Removing parent pages (core-concepts.md, customization-extension.md) may affect Just the Docs theme navigation.
    • Mitigation: Pages that previously had parent: frontmatter referencing deleted parent pages must have that field removed or updated. Test locally with bundle exec jekyll build.
Content Risks
  • Information loss during consolidation: Merging 14 pages into ~8 may drop useful content.
    • Mitigation: Each consolidation task must explicitly enumerate what content it absorbs from source pages. A review step should diff the old vs new to catch dropped content.
  • Inconsistent terminology across pages: If tasks are executed in parallel, different pages might use slightly different phrasings of the same canonical term.
    • Mitigation: The glossary page should be written first and referenced by all subsequent tasks. A final pass task should grep for retired terms.
Process Risks
  • Scope creep into code changes: The terminology overhaul may tempt changes to templates, skill files, or source code.
    • Mitigation: This plan is explicitly documentation-only. No files outside docs/ and AGENTS.md should be modified. The only exception is _config.yml for Jekyll metadata.
